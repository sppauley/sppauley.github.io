Why High-Throughput Computing?

When social scientists conduct research on increasingly large and complex datasets and perform increasingly complex analyses, computational resources naturally increase. Runtimes measured in hours and even days are commonplace.

Researchers are often forced to sit through these long runtimes. Those looking to expedite the process generally just throw more computational resources at the problem. However, this has several limitations. First, it can have a noticeable but nonetheless limited impact on performance. Second, these solutions typically centralize computing power in the hands of a few individuals at an institution. Even if researchers want to distribute access, this often takes the form of long and inefficient sign-up sheets.

High-throughput computing solves these problems in two ways.

First, it takes big, computationally demanding tasks and splits them into countless small jobs with fast runtimes. By running all these small jobs at once, high-throughput computing systems substantially reduce runtime.

During my research assistantship for the Center for Communication and Civic Renewal, I worked to process massive market research datasets with millions of rows and tens of thousands of columns. On my laptop, operations took around a day to complete. On a department machine with a top-end CPU, operations were reduced to around eight hours. Running hundreds of split jobs through CHTC reduced runtime to a total of about eight minutes.

Second, it improves the efficiency with which researchers can access computational resources. High-throughput computing systems like HTCondor manage clusters of resources like CPUs and GPUs — they allow users to submit jobs and efficiently assign them to the necessary resources.

As more institutions begin to use high-throughput computing systems like HTCondor — HTCondor reports hundreds of organizations using their system at the time of writing — it is worth leveraging these systems to tackle computationally demanding social science problems.


Working with High-Throughput Computing Systems

High-throughput computing has two major barriers to implementation in social science research: 1) the technical skills needed to work with the system and 2) knowledge of the data and code structures that will maximize high-throughput computing’s benefits.

This guide will primarily focus on that second category, since the technical steps will often vary based on institution. However, those interested in a thorough technical walkthrough can refer to my sample project and walkthrough here. Furthermore, the University of Wisconsin–Madison’s Center for High-Throughput Computing provides several resources for submitting jobs beyond what is covered in my sample project.

Understanding the structures that make for efficient high-throughput computing are necessary, however. When I started using HTCondor, even though I knew all the technical details for submitting jobs, my code still took forever to run. This was because I did not efficiently split my tasks into small, manageable jobs.

The remainer of this guide focuses on two methods for creating more efficient jobs: 1) dividing your data and 2) dividing your analysis.


Dividing Your Data

If you need to convert massive datasets or run analysis on subsets of the data that do not need to reference each other, you can divide your data into smaller chunks and create a job that runs the conversion or analysis scripts of those smaller chunks.

The principles of tidy data can inform this process. This structure has three major principles:

  • Each observation has its own row (ex. a survey respondent in a given year)

  • Each variable has its own column (ex. a respondent's age)

  • Each type of observational unit has its own table (ex. survey respondents vs social media posts)

Tidy data is useful because all data can essentially be treated the same, since the data contains only one type of observational unit (in this case, cities on a given day). Splitting the data can be done by dividing it into even chunks. This method scales well for large datasets, since you do not even need to read all the data into memory at the same time to divide it like this.

If your data structure isn't tidy, this becomes more difficult. Often, you need to develop custom code to split the dataset after reading it all into memory. These solutions are rarely uniform.

After you divide your data, the next step is to ensure that your code is reusable for multiple jobs with different datasets. This greatly increases the speed of writing your code and ensures you only have to debug your scripts in one place. Ideally, your code revolves around a file parameter that you can send to a set of resources through HTCondor and read/process in your scripts.

For an example of this in action, see my sample project.
